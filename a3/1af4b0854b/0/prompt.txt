現在いるディレクトリが Git リポジトリの場合、以下の手順で Pull Request を作成してください。

## 手順

1. **差分の確認**: `git status` と `git diff` でコミット対象の変更を確認
2. **コミット**: 未コミットの変更があればコミット
3. **プッシュ**: リモートにブランチをプッシュ
4. **PR テンプレートの読み込み**: リポジトリ内の `.github/PULL_REQUEST_TEMPLATE.md` を読み込む
5. **PR 作成**: `gh pr create` コマンドで PR を作成。テンプレートの形式に沿って、コミット内容を分析し PR の説明を記述する

## PR 説明の作成ルール

- PR テンプレートが存在する場合、そのセクション構造に従って記述する
- コミットログ (`git log origin/main..HEAD` または `git log origin/master..HEAD`) を分析し、変更内容を要約する
- 各セクションは簡潔かつ具体的に記述する
- テンプレートが存在しない場合は、以下の基本形式を使用:
  - **概要**: 変更の目的と内容
  - **変更点**: 主な変更のリスト
  - **テスト**: 動作確認の方法
- ドラフト状態で作成する

## PR 作成後

- 既に Pull Request が作られていた場合は、`gh pr view --web` で既存の PR を開く
- 新規作成した場合は、表示された URL を `open` コマンドでブラウザで開く

# PR Review Responder

PRのレビューコメントを確認し、対応要否を分類した上で指摘に対応してコードを修正する。

## ワークフロー

### 1. PR情報を取得

```bash
# ブランチ名
git branch --show-current

# リポジトリ情報（owner/repo を抽出）
git remote get-url origin
```

`git@github.com:owner/repo.git` または `https://github.com/owner/repo.git` から owner と repo を抽出する。

### 2. PRを特定

```bash
gh pr list --head <branch-name> --state open --json number,title,url
```

PRが見つからない場合はユーザーに報告して終了。
複数PRがある場合はユーザーに選択させる。

### 3. レビューコメントを取得

```bash
gh api repos/{owner}/{repo}/pulls/{pr_number}/comments --paginate
```

このAPIはインラインレビューコメント（コードに対するコメント）を返す。

### 4. コメントをフィルタリング

取得したコメントから以下を除外:
- `in_reply_to_id` が存在するもの（スレッドの返信。親コメントのみ対象）
- bot によるコメント（`user.type` が `Bot`）

各コメントについて、同じスレッドの返信も取得し、文脈を理解する。
返信の中で resolved 済みのマークがある場合はそのスレッドを除外する。

### 5. コメントを分類してユーザーに提示

各コメントを以下の3カテゴリに分類する:

| カテゴリ | 基準 | 例 |
|---------|------|-----|
| **対応する** | 明確なコード修正指摘 | 「この変数名を変更してください」「ここにバリデーション追加して」「typo: hoeg → hoge」 |
| **対応不要** | 質問・感想・賞賛・情報共有 | 「なぜこの実装にしたの？」「LGTM」「FYI: 別PRで対応予定」 |
| **要確認** | 意図が曖昧 or 大きな変更が必要 | 「このアーキテクチャ再考した方がよくない？」「パフォーマンス大丈夫？」 |

以下のフォーマットでユーザーに提示する:

```
## PR #123 レビューコメント分類

### 対応する（N件）
1. **file.rb:42** - 変数名を snake_case に変更 (@reviewer)
   > レビュー原文の引用
2. ...

### 対応不要（N件）
1. **file.rb:10** - 実装理由の質問（返信で回答済み） (@reviewer)
   > レビュー原文の引用

### 要確認（N件）
1. **file.rb:55** - アーキテクチャの再検討提案 (@reviewer)
   > レビュー原文の引用

上記の分類で対応を進めてよいですか？変更があれば指示してください。
```

ユーザーの承認を待つ。ユーザーが分類を変更した場合はそれに従う。

### 6. コード修正を実施

「対応する」に分類された（かつユーザーが承認した）コメントについて:

1. `path` でファイルを特定
2. `line`（diffの位置）と `original_line`（元ファイルの位置）で該当箇所を特定
3. `diff_hunk` で変更のコンテキストを確認
4. 該当箇所のコードを読み取り
5. 指摘内容に基づいてコードを修正
6. 修正後、構文エラーがないか確認する（lint / type check 等が利用可能なら実行）

### 7. 結果を報告

対応結果を以下のフォーマットで報告する:

```
## 対応結果

### 修正済み（N件）
1. **file.rb:42** - 変数名を snake_case に変更 ✓
   変更内容: `userName` → `user_name`
2. ...

### 要確認（N件）
（ユーザーの判断待ちのコメントがあれば再掲）
```

## 注意事項

- コメントのスレッド全体（返信含む）を読んで文脈を理解してから分類する
- `diff_hunk` の行番号と実際のファイルの行番号がずれている場合があるため、コードの内容で照合する
- 大きな変更（ファイル構成の変更、アーキテクチャ変更等）は「要確認」に分類し、自動で修正しない
