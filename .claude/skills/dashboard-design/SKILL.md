# Dashboard Design Skill

デジタル庁「ダッシュボードデザインの実践ガイドブック」(2024年5月)に基づくダッシュボード設計・実装ガイド。
ユーザーが「ダッシュボードを作りたい」「KPIを可視化したい」「データを見やすくしたい」等と言った場合にこのスキルを使う。

## ワークフロー

ダッシュボード作成は **要件の整理 → プロトタイピング → 実装** の3ステップで進める。

### Phase 1: 要件の整理

ユーザーに以下の5W1Hを確認する。未定の項目があれば質問して埋める。

| 問い | 確認事項 |
|------|--------|
| **Why** | ダッシュボードの最上位の目的。なぜ見るのか |
| **What, so What** | 目的達成のために知るべき情報、見た後にする判断や行動 |
| **Who** | 見る人の所属・立場・リテラシー |
| **When** | 見るタイミングと頻度 |
| **Where** | 提示する媒体・デバイス |
| **How** | 求められる機能・データ項目・更新頻度 |

**制約条件も確認する:**
- 情報の種類が多すぎないか → 減らす
- 伝える内容が複雑すぎないか → 複雑度を下げる
- データに欠損・更新頻度不足・分解能不足がないか

### Phase 2: プロトタイピング

1. **載せるべき情報を一覧化**: 指標の分類・情報の詳細・補足情報・グラフ候補をテーブルで整理
2. **骨格レイアウトを設計**: 情報の優先度と関係性に基づき配置を決定
3. **モックアップを作成**: グラフを配置し最終イメージに近い図を作る

**載せるべき情報を選ぶ4原則:**
- 目的に則する: 目的に当てはまるデータだけ選び、それ以外は取り除く
- 分解できる: 全体から部分へ、読み解ける単位まで分割
- 違いに気づける: 突出するデータや変化の時期と傾向に気づける
- 鮮度が高い: 最新データを反映し、定期更新の仕組みを構築

### Phase 3: 実装とチェック

実装後、以下のチェックリストで最終確認する。

## ダッシュボード設計の4原則

1. **構造を伝える**: 全体→部分の階層構造。最初に全体を示し、詳細に辿り着けるレイアウト
2. **適切な情報量にする**: 過多な情報で利用意欲を削がない。見る人の負担にならない総量
3. **複雑な操作を要求しない**: 操作なし or 少ない操作回数で必要な情報が得られる
4. **比較対象を提供する**: 目標値・前年値・前年比など、意思決定に必要な比較軸を併記

## レイアウト設計

- **視線の流れ**: 左上→右下（Z型）。全体→部分の順で配置
- **左上に最も重要な指標**を大きく配置。右下に向かって詳細情報
- **背景色の差**で情報の重要度を表現。左上が最も目を引くように
- **フィルターは画面の上部か左部**に配置。影響を受ける情報はその下か右に
- **レイアウトグリッド**: 16:9画面で縦横6分割のグリッドを使用（1920x1080px推奨）

### レイアウトパターン

- **単一指標を複数角度で提示**: 左列に指標値、右に構成比・分類別・時系列を配置
- **複数指標を併記**: 上段にA/B/Cの指標値、中段に分解、下段に時系列
- **絞り込み型**: 左にフィルター・時系列、右に数値テキスト・詳細表
- **表中心型**: 左にフィルター、右に数値とデータ表

## グラフの選び方

| グラフ | 値の表現 | 伝える情報 | 使い分け |
|--------|---------|----------|---------|
| **折れ線グラフ** | 位置 | 時間変化・傾向 | 横軸は時間。系列名は折れ線近傍に配置 |
| **棒グラフ** | 長さ | 数量比較 | 分類毎の比較。集合棒・積み上げ棒あり |
| **面グラフ** | 位置・面積 | 時間変化+構成比 | 積み上げ面で構成比の時間変化を表現 |
| **円グラフ** | 角度・面積 | 構成比 | 全体の総量が明らかな場合のみ。多くの場合は棒グラフの方が正確 |
| **指標（数値）** | - | 端的な状況把握 | KPI値、達成率など。最も目立つ位置に配置 |
| **表** | - | 詳細データ一覧 | 多次元データの参照用。色や図形で補足可 |

## カラーパレット

デジタル庁デザインシステムに基づく配色ルール:

**テキスト色:**
- Body: `#1A1A1A` / Description: `#626264` / On Fill: `#FFFFFF` / Link: `#0017C1`
- Positive Number: `#0031D8` / Negative Number: `#FA0000`

**背景色:**
- Primary: `#0031D8` / Secondary: `#F8F8FB`
- Positive: `#E8F1FE` / Negative: `#FDEEEE`

**グラフステータス色:**
- Positive Bar: `#0031D8` / Negative Bar: `#FA0000`
- Positive Data Bar: `#C5D7FB` / Negative Data Bar: `#FFBBBB`

**グラフの配色ルール:**
- 1〜3色 → カラーパレットの太字明度（メイン明度）から選択
- 4〜5色 → 全明度から選択
- **背景とグラフ色面のコントラスト比は3:1以上**を確保
- 3:1未満の場合、色面領域近くに数値を記載するか、ホバー/フォーカスで表示
- 数値の色は背景に対して4.5:1以上のコントラスト比

**色覚多様性への配慮:**
- 色のみで分類を識別しない。データ点の形やグラフの塗りの模様でも識別可能にする
- ユニバーサルデザイン推奨の配色、色覚特性シミュレーションを活用

## グラフ設計の原則

### 知りたいことを知れる
- **シンプルにする**: 冗長なデータや説明、過度に装飾的な表現を取り除く
- **意味のある順列にする**: 数量の大小、アルファベット順、更新日順など情報に適した並び順
- **強弱をつける**: 重要な情報には色・サイズ・太字で視覚的な差をつける
- **待たせすぎない**: 画面表示時間や操作反応を短く

### 誤解を生まない
- **わかりやすく表記**: 前提知識不要な表現。タイトル・系列名・単位を丁寧に
- **データを定義する**: 何を対象に収集したデータで、数値の意味は何か、いつ更新されたか
- **表現を歪曲しない**: 軸の値範囲を恣意的に設定しない。値の範囲を省略しない
- **メタ情報を記載する**: データソース・更新日・注釈・免責事項を提供

## Do's & Don'ts

### Do（望ましい表現）
- 全体の指標と詳細のグラフを表示する（指標→時系列→分解の順）
- グラフ項目の並び順に意味を持たせる（数量の大小順、更新日順など）
- 不要な要素は削除する（目盛線・冗長タイトル・小数部を最小限に）
- 不要な装飾やリッチな表現は使わない（3D・ドロップシャドウ禁止）
- グラフに使用する色数を絞る（1〜5色。注目すべき系列を明確に）
- タイトルにグラフの内容とデータ種別を表記する（例: 「国産自動車の出荷台数（月次推移）」）
- タイトルや凡例をシンプルに保つ（端的で簡潔な表現）
- グラフと凡例を隣接させる（凡例の順序をグラフと一致させる）
- グラフの原点は原則として0にする（棒グラフは特に）
- 色のみで分類を識別しない（形状・模様でも識別できるように）

### Don't（望ましくない表現）
- 詳細の要素だけが並んでいる（全体感を示す指標がない）
- 意味のない順序で並んでいる（あいうえお順で数量比較する棒グラフなど）
- 目盛線やタイトル、小数部などが必要以上に記載されている
- 3D表現やドロップシャドウなど数値と無関係な装飾
- 色数が多すぎて注目すべきグラフがわからない
- 何を表すグラフかすぐにわからないタイトル（例: 「国産自動車」だけ）
- 端的でないタイトル（冗長な文章のタイトル）
- 凡例がグラフと離れている、または凡例の順番がグラフと逆
- 原点が0でない棒グラフ（数量の差が誇張・矮小化される）
- 違うグラフで同じ形のマーカーを使っている

## チェックリスト

実装完了後、以下を確認する:

### 利用者の体験
- [ ] 利用者が判断や行動に必要な情報が提供できているか？
- [ ] データの全体像を捉えるための基準となる指標が掲載されているか？
- [ ] 本当に必要な情報だけに絞り込めているか？ほとんど使わない情報が掲載されていないか？
- [ ] 利用者に必要以上の操作を求めていないか？数回の操作で目的が達成されているか？
- [ ] 利用者が迷わず必要な情報まで辿り着けるか？
- [ ] 複数の利用者からのフィードバックを得て改善を重ねたか？

### 指標と表、グラフのデザイン
- [ ] 利用者が見たいと思っている数値へ優先的に目が向くようになっているか？
- [ ] グラフや表の配列順序が認知しやすいものになっているか？
- [ ] グラフと凡例が隣接され、対応づけがわかりやすくなっているか？
- [ ] 他のデータとの比較や時間的な変化によって、すぐに違いを気づけるようになっているか？
- [ ] グラフの色数が1〜5色程度に絞れているか？
- [ ] グラフに含まれる要素（グリッドや枠線）を最小限にしているか？
- [ ] 不要な装飾や画像、3Dやシャドウのようなリッチな表現は避けているか？
- [ ] 誰が見てもわかりやすく、誤解のないタイトルや系列名になっているか？
- [ ] 棒グラフの原点は0になっているか？
- [ ] データの更新日は記載されているか？

### データ設計
- [ ] 利用者の目的に見合ったデータだけが選ばれているか？
- [ ] 利用者が見たいレベルまでデータを分解できるようになっているか？
- [ ] 利用者が必要な頻度でデータが更新されるような仕組みや運用プロセスになっているか？
- [ ] 数量単体では判断が困難な場合、理解を促すために比較できる数値が掲載されているか？
- [ ] データ定義を参照できるようになっているか？

### テクニカル
- [ ] 利用者が実際に見る環境やデバイスで問題なく表示され、動作することを確認したか？
- [ ] ダッシュボードの読み込み速度が遅すぎないか？

### アクセシビリティ
- [ ] 色だけでグラフの情報を識別していないか？
- [ ] グラフに代替テキスト（alt情報）を付与しているか？
- [ ] データファイル（ExcelやCSV）を公開しているか？
- [ ] ダッシュボードの情報が要約されたテキストを公開しているか？（一般公開の場合）

## 実装時の技術指針

- フレームワークの指定がなければ、ユーザーのプロジェクトに合わせて選定する
- グラフライブラリ例: Recharts, Chart.js, D3.js, ECharts, Plotly など
- レスポンシブ対応: 16:9画面（1920x1080px）を基準に設計し、モバイル対応も考慮
- パフォーマンス: 大量データの場合はページネーション・仮想スクロール・遅延読み込みを検討
